# 猫超月账单整理升级设计

## 目标

在现有“猫超账单整理”能力基础上升级生成结果文件 `猫超{month}月账单数据表格.xlsx`，新增以下能力：

- 新增 `对账单列表` sheet
- 新增 `万相台推广数据表格` sheet
- 新增 `智多星推广数据表格` sheet
- 在 `开票表` 右侧自动生成利润汇总区域

升级后仍保持当前命令入口、账单下载方式、归档目录结构和已有主表结构稳定。

## 已确认约束

- 所有新增数据必须走已有 CLI，不允许在业务层新增 requests、curl 重放、登录逻辑或抓包逻辑
- 优先复用现有 `Ops-Cli`
- 新增 sheet 需要先删后建，不影响已有主表和已有公式
- 新增原始数据 sheet 保持原字段顺序和字段名不变，只允许做轻量格式标准化
- 利润汇总位置采用“方案 A”：从 `开票表` 当前最大列右侧空 2 列开始
- 工作簿内已存在并继续使用 `开票表` 和 `成本表`
- `实际成本` 优先读取 `成本表`，缺失时才退回 `成本 × 商品数量`

## 数据流设计

升级后的执行流如下：

1. 现有流程合并 HDB 数据，生成主表、`货款表格`、`票扣表格`、`账扣表格`、`开票表`、`成本表`
2. 账单下载结果中获取或补充获取 `对账单列表`
3. 通过推广账单 CLI 下载 `万相台` 和 `智多星` 原始文件
4. 打开同一个 workbook，对新增 sheet 做“同名先删后建”
5. 写入三个新增原始数据 sheet
6. 在 `开票表` 右侧写入利润汇总区域
7. 保存工作簿并继续复用现有归档流程

关键要求：

- workbook 只加载一次，避免重复 IO
- 新增 sheet 和利润表都在同一份 workbook 对象上完成
- 先构建现有业务 sheet，再补充新增 sheet 和利润汇总

## 工作簿写入设计

新增公共写入约定：

- 若目标 sheet 已存在，删除后重新创建
- 第一行写表头，后续写原始行
- 轻量格式标准化只包含：
  - 自动列宽
  - 将明确可识别的数值写为数值单元格
  - 将明确可识别的日期写为日期单元格或保持原始文本
- 不改变字段名
- 不重排字段
- 不新增业务字段到原始数据 sheet

## 利润汇总设计

### 区域定位

基于 `开票表`：

- 起始列 = `开票表.max_column + 2`
- 起始行默认使用 `5`
- 若 `开票表` 数据行不足以使区域观感合理，可允许后续实现加入“最小值 5 + 轻量居中”策略，但首版优先固定在第 5 行附近

本次实现默认规则先明确为：

- `summary_start_col = invoice_sheet.max_column + 2`
- `summary_start_row = 5`

这样简单稳定，可复现，也满足“5 行附近”的要求。

### 表格结构

三列固定：

1. 项目名称
2. 台数
3. 含税金额

行结构固定：

1. `{month}月份利润表 | 台数 | 含税金额`
2. `销售金额（开票金额）`
3. `实际成本`
4. `营销推广`
5. `票扣`
6. `账扣`
7. `最终利润`

其中只有 `销售金额（开票金额）` 填写台数，其余台数列为空。

### 计算规则

#### 台数

来源：

- `开票表`

规则：

- 根据表头名查找 `商品数量`
- 全列求和
- 写入整数值

#### 销售金额（开票金额）

来源：

- `开票表`

规则：

- 根据表头名查找 `开票金额`
- 全列求和
- 保留两位小数

#### 实际成本

优先来源：

- `成本表`

规则：

- 根据表头名查找 `金额`
- 全列求和

回退来源：

- `开票表`

回退规则：

- 查找 `成本` 和 `商品数量`
- 按行执行 `成本 × 商品数量`
- 汇总求和

#### 营销推广

来源一：

- `万相台推广数据表格`

规则：

- 根据表头名查找 `收支类型` 和 `金额`
- 筛选 `收支类型 == 支出`
- 对金额求和

来源二：

- `智多星推广数据表格`

规则：

- 根据表头名查找 `类型`
- 金额列按候选名称自动识别，优先级按代码内可配置候选列表实现：
  - `金额`
  - `收支金额`
  - `发生金额`
- 筛选 `类型 == 从冻结中转出`
- 对筛选结果的金额列求和

最终规则：

- 营销推广 = 万相台支出合计 + 智多星从冻结中转出合计

#### 票扣

来源：

- `开票表`

规则：

- 根据表头名查找 `票扣`
- 全列求和
- 输出正数绝对值

#### 账扣

来源：

- `账扣表格`

规则：

- 根据表头名查找 `含税金额`
- 全列求和
- 输出正数绝对值

#### 最终利润

规则：

- `销售金额（开票金额） - 实际成本 - 营销推广 - 票扣 - 账扣`

### Decimal 与异常值处理

所有金额计算统一使用 `Decimal`，转换规则必须兼容：

- `None`
- 空字符串
- 含逗号的数字字符串
- 普通字符串数字
- 已经是 int / float / Decimal 的值

对于无法识别的值：

- 不抛出格式异常中断整个任务
- 按空值处理
- 保留日志或统计信息供排查

### 样式设计

利润汇总区域样式固定如下：

- 3 列结构
- 全部单元格加细边框
- 标题行加粗
- 第一列浅黄色底色
- 金额列右对齐
- 台数字段居中或整数格式
- 金额列使用两位小数格式
- 最终利润整行加粗，至少保证标签和金额加粗
- 整体布局紧凑

不做复杂合并单元格，以稳定输出和后续维护为优先。

## CLI 集成设计

业务层不直接请求平台。

统一通过已有 `run_ops_json()` 调用：

- 对账单列表：`--json tmcs bill download --start/--end 或 --last-month --download-statement-list`
- 万相台：`--json tmcs promotion-bill download --source wxt --start/--end 或 --last-month`
- 智多星：`--json tmcs promotion-bill download --source zdx --start/--end 或 --last-month`

编排层需要做两件事：

- 从 CLI 返回 JSON 里提取下载文件路径
- 对下载文件做存在性校验，并将产物接入 sheet 写入流程

## 错误处理

需要明确报错，不做静默吞错。

关键错误场景：

- CLI 返回失败
- CLI 成功但未返回文件路径
- 文件路径存在但文件不存在
- 原始 sheet 字段缺失
- `开票表` / `成本表` / `账扣表格` 缺失

报错要求：

- 明确指出是哪个数据源、哪个 sheet、哪个字段有问题
- 尽量保持任务整体退出信息可读
